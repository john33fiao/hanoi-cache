# CLAUDE.md

이 저장소의 에이전트 작업 기준 문서입니다.
`AGENTS.md`와 `GEMINI.md`는 이 문서를 추종합니다.
지침이 충돌하면 `CLAUDE.md`를 우선합니다.

## 목적

이 저장소는 하노이 권역의 지오코딩과 날씨 조회를 제공하는 작은 Rust HTTP 서버입니다.
에이전트는 현재의 단순한 구조를 유지하면서, 필요한 변경만 정확히 수행해야 합니다.

## 먼저 읽을 문서

작업 전에 아래 문서를 먼저 확인합니다.

- `README.md`: 실행 방법, 환경 변수, API 사용 예시
- `docs/architecture.md`: 요청 흐름, 캐시 정책, 외부 공급자, 설계 제약

API 의미나 사용자 사용법이 바뀌면 위 두 문서도 함께 갱신합니다.

## 저장소 개요

- `src/main.rs`: 앱 시작점, 라우터, 환경 변수 로딩
- `src/handlers.rs`: `/geocode`, `/weather`, `/weather/{loc}` 핸들러와 캐시 사용
- `src/providers.rs`: Nominatim, Open-Meteo, OpenWeatherMap 호출
- `src/cache.rs`: 인메모리 캐시
- `skills/rtd-before/SKILL.md`: 구현 전 계획/범위/검증 게이트
- `skills/rtd-after/SKILL.md`: 구현 후 회귀/품질/READY 게이트

## 작업 원칙

- 문서와 사용자 대상 설명은 한국어로 유지합니다.
- 현재 구조를 존중합니다. 불필요한 추상화, 과도한 계층 분리, 성급한 일반화는 피합니다.
- 이 서비스의 핵심 제약을 먼저 고려합니다.
  - `/weather/{loc}`는 고정된 위치 슬러그만 지원합니다.
  - `/weather?latitude=<f64>&longitude=<f64>`는 GPS 좌표를 지원합니다.
  - `/weather`에서 좌표가 없거나 잘못되면 기본 위치 `hoankiem`으로 처리합니다.
  - 날씨 응답은 `current`와 `daily`를 함께 포함합니다.
  - 캐시는 인메모리이며 프로세스 재시작 시 사라집니다.
  - 서버 시작 시 `OPENWEATHERMAP_API_KEY`가 필요합니다.
- API 동작을 바꾸면 코드만 바꾸지 말고 `README.md`와 `docs/architecture.md`까지 같이 맞춥니다.
- 검증 가능한 사실만 적습니다. 없는 파일, 없는 테스트, 없는 실행 경로를 가정하지 않습니다.

## 기본 작업 흐름

1. 관련 코드와 문서를 읽고 변경 범위를 명확히 합니다.
2. 파일을 수정하기 전에 가능하면 `rtd-before` 스킬을 사용해 계획을 닫습니다.
3. 최소 변경으로 구현합니다.
4. 구현 후 `rtd-after` 스킬로 회귀와 품질을 다시 점검합니다.
5. 가능한 검증을 수행하고, 실행하지 못한 검증은 이유와 리스크를 명시합니다.

## 로컬 스킬

현재 이 저장소에는 아래 두 스킬이 추가되어 있습니다.

### 1. `rtd-before`

- 경로: `skills/rtd-before/SKILL.md`
- 목적: 코드 또는 문서 수정 전에 계획 품질, 범위, DoD, 가정, 테스트 전략, 롤백 경로를 먼저 정리하는 사전 게이트
- 사용 시점: 기능 추가, 버그 수정, 문서 수정, 리팩터링, API 변경 전
- 기대 결과:
  - What / In-Out / DoD / 제약 정리
  - 변경 파일 예상
  - 테스트 전략
  - 롤백 전략
  - 과도한 설계 제거

주의:

- 스킬 내용에 현재 저장소에 없는 파일이나 흐름이 언급될 수 있습니다.
- 예를 들어 `docs/API_Doc.md` 같은 기준 문서가 실제로 없으면, 있다고 가정하지 말고 "없음"을 명시한 뒤 현재 저장소 기준으로 진행합니다.
- 스킬의 핵심 의도는 "수정 전에 계획과 범위를 먼저 닫는 것"입니다.

### 2. `rtd-after`

- 경로: `skills/rtd-after/SKILL.md`
- 목적: 구현 후 목적 적합성, 회귀, 보안, 정리 상태, READY 여부를 검토하는 사후 게이트
- 사용 시점: 구현 완료 후 커밋 전, PR 전, 결과 정리 전
- 기대 결과:
  - DoD 충족 여부 점검
  - 잠재 버그와 회귀 확인
  - 불필요한 코드와 중복 정리
  - 테스트/검증 근거 정리
  - 최종 READY / NOT READY 판단

주의:

- 저장소에 테스트나 CI가 없다면 있는 척하지 않습니다.
- 실행 가능한 검증이 제한적이면 그 한계를 그대로 적고, 남은 리스크를 명시합니다.

## 구현 시 권장 검증

변경 성격에 따라 가능한 범위에서 아래를 사용합니다.

- 포맷: `cargo fmt`
- 정적 검증: `cargo check`
- 테스트: `cargo test`
- 수동 확인: `cargo run` 후 `/geocode`, `/weather/{loc}`, `/weather?latitude=...&longitude=...`, `/weather` 호출

실행이 불가능하거나 불필요하면 이유를 짧게 남깁니다.

## 에이전트 문서 동기화 규칙

- 에이전트 지침을 바꿀 때는 `CLAUDE.md`를 먼저 수정합니다.
- `AGENTS.md`, `GEMINI.md`는 `CLAUDE.md`를 요약한 추종 문서로 유지합니다.
- 세 문서 간 내용이 어긋나면 `CLAUDE.md`가 기준입니다.
