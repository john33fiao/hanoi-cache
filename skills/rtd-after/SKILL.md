# RTD After

구현이 끝난 변경을 배포 가능한 수준까지 끌어올리는 저장소 전용 사후 게이트입니다.
이미 diff가 있는 상태에서 목적 적합성, 회귀, 보안, 정리, 최종 READY 판정을 수행하십시오.

## 시작 전 확인

- 구현 대상과 범위는 `rtd-before` 또는 동등한 계획 문서로 정리되어 있어야 합니다.
- 최소한 리뷰 가능한 diff 또는 변경 파일 집합이 있어야 합니다.
- 저장소에 빌드/테스트 경로가 없으면 없는 사실을 그대로 적고, 문서 일관성 및 가정 명시 중심으로 검토하십시오.
- API 의미를 바꾸는 작업이면 `docs/API_Doc.md`를 기준 문서로 사용하십시오.

## 작업 순서

### 1. 구현이 목적에 맞는지 검토하십시오

- DoD 항목별 충족 여부를 확인하십시오.
- PASS 기준: DoD가 모두 충족되거나 합리적 보완 방안이 제시됩니다.

### 2. 잠재적 버그, 크리티컬 이슈, 보안 문제를 검토하십시오

- 입력 검증, 인증/인가, 인젝션, 경로/파일, 직렬화, 경쟁 상태, 리소스 누수, 비밀값 노출을 점검하십시오.
- 의존성 변경이 있으면 보안/라이선스 리스크를 확인하십시오.
- PASS 기준: 고위험 이슈가 없거나 모두 완화됩니다.

### 3. 회귀가 없는지 검토하십시오

- 기존 동작과의 호환성을 점검하십시오. API, 문서, 설정, CLI, UX가 핵심입니다.
- 필요한 테스트 추가 또는 수정을 수행하십시오.
- PASS 기준: 회귀가 없거나 발견된 회귀가 모두 해결됩니다.

### 4. 큰 함수나 파일을 적절히 나누십시오

- 과도한 책임, 긴 함수, 거대 파일은 분할하십시오.
- PASS 기준: 단일 책임이 읽히는 단위로 정리됩니다.

### 5. 기존 코드와 통합 또는 재사용할 수 있는 부분을 검토하십시오

- 중복 유틸, 도우미, 도메인 로직이 있으면 재사용으로 치환하십시오.
- PASS 기준: 중복이 제거되고 결합이 자연스럽습니다.

### 6. 사이드 이펙트가 없는지 확인하십시오

- 성능, 캐시, 트랜잭션, 이벤트/큐, 전역 상태, 설정 플래그, 로깅, 메트릭 영향을 점검하십시오.
- PASS 기준: 부작용이 없거나 문서화/완화/테스트로 통제됩니다.

### 7. 전체 변경 사항을 다시 검토하십시오

- diff를 처음 보는 리뷰어 관점으로 읽으십시오.
- PASS 기준: “왜 필요한 변경인지”가 diff만으로 납득됩니다.

### 8. 불필요해진 코드를 정리하십시오

- 죽은 코드, 임시 플래그, 사용되지 않는 함수/파일, 디버그 로그를 제거하십시오.
- PASS 기준: 잔재가 남지 않습니다.

### 9. 코드 품질 기준을 확인하십시오

- 레포 표준의 포맷, 린트, 타입체크, 테스트, CI 수준에 맞추어 실행하거나 동일 수준으로 정렬하십시오.
- 현재 저장소처럼 실행 경로가 제한적이면 그 한계를 명시하고 가능한 검증 근거를 남기십시오.
- PASS 기준: 실패 요소가 없거나 미실행 사유와 잔여 리스크가 명확합니다.

### 10. 사용자 흐름에서 문제는 없는지 확인하십시오

- 사용자 시나리오 1~3개를 실제 흐름으로 따라가며 확인하십시오.
- PASS 기준: 주요 흐름이 막힘 없이 동작합니다.

### 11. 발견된 문제를 끝까지 파고들어 재검토하십시오

- “근본 원인 -> 영향 범위 -> 재발 방지” 순서로 처리하십시오.
- PASS 기준: 임시 봉합이 아니라 원인과 영향 범위가 닫힙니다.

### 12. 최종 READY 여부를 판정하십시오

- DoD 충족
- 테스트/린트/빌드 기준 충족
- 보안 고위험 0
- 회귀 0
- 문서/변경 로그 반영
- 롤백/마이그레이션 경로 명시
- PASS 기준: READY 판정 가능. 아니면 NOT READY와 블로커를 명시합니다.

### 13. 커밋과 PR 초안을 정리하십시오

- 레포의 커밋 관습이 있으면 따르십시오.
- 커밋은 가능하면 논리 단위로 나누십시오.
- PR 초안에는 요약, 배경, 변경점, 테스트 방법, 리스크/롤백, 영향 범위를 포함하십시오.

## 출력 형식

- 각 단계는 아래 형식으로 기록하십시오.
- ` [Step N] 제목 - PASS/FAIL `
- 핵심 근거는 최대 5줄로 제한하십시오.
- 필요할 때만 변경/조치 목록과 다음 단계 리스크를 덧붙이십시오.

## 종료 조건

- READY면 커밋/PR 단계로 진행하십시오.
- NOT READY면 블로커와 남은 리스크를 먼저 정리하고 종료하십시오.
